# 主线方案说明

## 1. 当前统一主线

当前项目后续论文、汇报和实验，统一以 `physics_subspace_zoo_v1` 作为正式主线。

统一这条主线的原因很直接：

- 它已经在当前代码状态下完成了全量复验；
- 它同时具备较高 ASR 与较低查询开销；
- 它的方法叙事比纯搜索更容易说服审稿人，因为核心更新步骤建立在子空间 PG-ZOO 上。

当前复验结果为：

| system | subset size | ASR | avg queries |
| --- | ---: | ---: | ---: |
| case14 | 1717 | 98.19% | 13.53 |
| case118 | 1795 | 90.53% | 55.53 |

- `results/misc/misc/verify_physics_subspace_full_v2/runs/ieee_case118_verify_physics_subspace_full_v2/attack_summary.json`

## 2. 主线叙事

这条主线建议统一表述为：

**物理先验引导的一次性子空间黑盒 PG-ZOO 攻击框架**

更具体地说，它的三层逻辑是：

1. 用物理先验压缩状态搜索空间；
2. 在压缩后的子空间内做零阶梯度估计；
3. 在保证 FDIA 语义不被破坏的前提下，生成可直接注入的对抗修正。

这条叙事的优点是：

- 不是纯搜索；
- 不是完全依赖代理模型；
- 物理信息的作用边界清楚，即“先压缩维度，再在子空间内查询优化”。

## 3. 两个系统上的实现形式

虽然两个系统的具体子空间实例化不同，但上层逻辑完全一致。

### 3.1 case14

`case14` 采用二维局部区域子空间：

- `attack_mode = region_search`
- `region_size = 2`
- `region_candidates = 1`
- `state_subspace_pgzoo = True`

含义是：

- 先用物理先验选一个二维局部区域；
- 再只在该二维区域内做 PG-ZOO 更新。

### 3.2 case118

`case118` 采用固定 top-64 support：

- `attack_mode = single_active_region_search`
- `single_active_size = 117`
- `state_basis_dim = 12`
- `state_subspace_pgzoo = True`

这一步的含义是：
- `case118` 不再使用过强的 top-k support 硬截断；
- 保留全状态 support，但通过物理基底把实际零阶优化限制在 12 维子空间内。

含义是：

- 先把全状态空间一次性收缩到 top-64 support；
- 再只在该 support 内做 PG-ZOO。

## 4. 与旧主线的关系

`deterministic_mainline_v1` 现在应作为参考基线，而不是正式主线。

原因是：

- 它的核心仍然更偏“确定性搜索”；
- 尽管结果不错，但在论文表达上更容易被认为是 pattern search 的工程调优；
- 相比之下，`physics_subspace_zoo_v1` 更容易形成“物理先验 + 子空间零阶优化”的规范方法论。

因此，后续建议的角色划分是：

- `physics_subspace_zoo_v1`：正式论文主线
- `deterministic_mainline_v1`：强基线
- 其它实验分支：探索线，不写进主结果表

## 5. 当前可以支撑到什么程度

如果目标是：

- 硕士毕业论文：已经足够作为主线；
- 一般 SCI 或 IEEE 应用类论文：基本可以；
- `TSG / TII / IoTJ` 级别：还不够，需要继续补强创新性与理论性。

当前不足主要有三点：

- “为什么这种物理收缩最合理”还缺少更系统的实证支撑；
- `case118` 的查询长尾仍然存在；
- 查询复杂度优化的理论表达还不够强。

## 6. 后续工作原则

从现在开始，所有后续工作都按下面的原则推进：

1. 不再同时维护多条正式主线；
2. 一切新增实验都优先与 `physics_subspace_zoo_v1` 对齐；
3. 新实验若不能在全量或大样本验证中稳定成立，只保留为探索结论，不进入主文稿；
4. 后续增强重点放在：
   - 查询复杂度优化；
   - 物理先验有效性证明；
   - 面向论文的消融与对比实验。
