# 精炼消融总结（2026-03-28）

## 1. 本轮消融的目的

本轮不再继续扩展新的攻击分支，而是围绕当前正式主线 `physics_subspace_zoo_v1`，补齐论文最需要的三组精炼消融：

1. `case14`：验证物理区域收缩是否必要；
2. `case118`：验证旧版 `hard support-64` 是否真的是有效设计；
3. `case118`：验证 FDIA 语义保持项是否必要，以及它的真实作用是什么。

## 2. 消融结论

### 2.1 case14：物理区域收缩是核心收益来源

对比：
- 主线：`ASR=98.19%`，`avg_queries=13.53`，`p95=39`
- 去掉区域收缩：`ASR=79.85%`，`avg_queries=26.73`，`p95=83`

结论：
- `case14` 上的主收益确实来自局部物理区域选择；
- 一旦不做区域收缩，成功率明显下降，查询成本显著上升；
- 因此 `case14` 的论文叙事可以明确写成“物理区域收缩 + 子空间 PG-ZOO”。

### 2.2 case118：旧版 hard support-64 是错误设计

对比：
- 旧版 `hard support-64`：`ASR=87.91%`，`avg_queries=66.83`，`p95=243`
- 新主线 `full support + physical subspace`：`ASR=90.53%`，`avg_queries=55.53`，`p95=243`

结论：
- 在 `case118` 上，过强的 support 硬截断会同时损害成功率和平均查询效率；
- 物理先验在大系统中不应被解释为“强制删掉大量状态维度”；
- 更合理的做法是保留全 support，再利用物理基底把搜索压缩到低维子空间中。

因此，`case118` 的论文叙事应改写为：
- 全 support 上的物理子空间 PG-ZOO；
- 而不是 top-k support 截断。

### 2.3 case118：FDIA 语义保持项是轻量正则，而不是主成功率来源

对比：
- 主线：`ASR=90.53%`，`avg_queries=55.53`
- 去掉语义保持：`ASR=90.14%`，`avg_queries=55.95`

同时语义指标变化为：
- `state_projection_ratio_mean`：`0.132 -> 0.129`
- `measurement_projection_ratio_mean`：`0.140 -> 0.141`
- `state_offsupport_energy_ratio_mean`：`0.138 -> 0.141`

结论：
- 语义保持项并不是当前主成功率提升的主要来源；
- 它的作用更接近“轻量正则”，用于稳定 FDIA 骨架的一致性；
- 因此论文中不应该把它写成决定性创新点，而应写成辅助约束项。

### 2.4 是否可以把 case14 也统一成“全 support + 物理子空间 PG-ZOO”

本轮额外验证了统一版方案，即在 `case14` 上也不做局部区域收缩，而是改成：
- `attack_mode = single_active_region_search`
- `single_active_size = 13`
- `region_size = 13`
- 在全状态 support 上直接做物理子空间 PG-ZOO

尝试过的典型配置包括：
- `state_basis_dim = 4`：`ASR=81.25%`，`avg_queries=25.78`（256 样本）
- `state_basis_dim = 6`：`ASR=80.08%`，`avg_queries=30.59`（256 样本）
- `state_basis_dim = 2`：`ASR=90.62%`，`avg_queries=17.33`（256 样本）

其中最好的统一版是 `state_basis_dim = 2`，继续做全量验证后结果为：
- `ASR=89.63%`
- `avg_queries=16.98`

而当前正式主线在 `case14` 上的全量结果为：
- `ASR=98.19%`
- `avg_queries=13.53`

结论：
- `case14` 上虽然也能勉强使用“全 support + 物理子空间 PG-ZOO”，但性能明显弱于当前“物理局部区域收缩 + PG-ZOO”；
- 因此**从论文结果和方法有效性角度，不建议为追求形式统一而强行统一 case14 与 case118 的实现**。

## 3. 当前主线应如何表述

基于本轮消融，当前主线应统一表述为：

- `case14`：物理区域收缩 + 子空间 PG-ZOO
- `case118`：全 support 上的物理子空间 PG-ZOO

两者共同点是：
- 物理信息不直接代替黑盒查询；
- 物理信息用于定义更合理的低维搜索坐标系；
- 黑盒查询负责在该坐标系内完成零阶优化。

两者差异在于：
- `case14` 的最优方式是“先用物理信息做强局部收缩，再在局部子空间中优化”；
- `case118` 的最优方式是“保留全 support，再用物理信息构造低维物理基底”。

因此，当前更合理的说法不是“两个系统完全同一实现”，而是：
- **同一方法论**
- **不同系统规模下采用不同强度的物理收缩方式**

## 4. 论文写作建议

建议在论文中按如下方式组织这组消融：

1. 先用 `case14` 说明“小系统中，物理区域收缩可以显著减少无效搜索”；
2. 再用 `case118` 说明“大系统中，不能把物理先验误用为过强的 hard support 截断”；
3. 最后用语义保持消融说明“该项是稳定语义一致性的轻量正则，而不是主要收益来源”。

这样写有两个好处：
- 逻辑更闭环，不会把所有收益都模糊归因于“物理先验”；
- 创新点边界更清楚，更容易说服审稿人。
